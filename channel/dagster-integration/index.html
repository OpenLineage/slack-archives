Hello, my team would like to contribute to the OpenLineage-Dagster integration work and wanted to start a public channel for general discussion on this topic.

Issue #489 is currently open for review and includes the proposal for the integration. As we proceed with the initial implementation, we’d appreciate feedback from the community to make sure the approach is reasonable and OpenLineage events are captured accurately.

Looking forward to more discussions. Thanks!

<https://github.com/OpenLineage/OpenLineage/issues/489|#489 [INTEGRATION][Dagster] Openlineage-Dagster Integration>
<p>Hello, my team would like to contribute to the OpenLineage-Dagster integration work with the following proposal.</p> <p><strong>Integration Target:</strong></p> <p><a href="https://github.com/dagster-io/dagster">Dagster</a></p> <p><strong>Proposal:</strong></p> <p>New <em>openlineage-dagster</em> library that integrates Dagster with OpenLineage for automatic metadata collection</p> <p><strong>Incremental Support:</strong></p> <ol><li>Initial integration handling basic metadata (job runs)</li><li>Individual “extractor” handling additional metadata (input/output datasets, data quality metrics, and etc.)</li> </ol> <p><strong>Proposed Implementation:</strong></p> <p>Dagster uses an <a href="https://docs.dagster.io/deployment/executors#executors"><code>Executor</code></a> to <a href="https://docs.dagster.io/_apidocs/internals#dagster.Executor.execute"><code>execute()</code></a> Dagster job (aka pipeline) runs, and each execution yields a stream of <a href="https://docs.dagster.io/_apidocs/execution#dagster.DagsterEvent"><code>DagsterEvents</code></a> from different stages of the run.</p> <p>As Dagster supports custom Executor, we can implement a custom <code>OpenLineageExecutor</code> that will:</p> <p>• act as a thin wrapper over an existing implementation of Executor • translate given <a href="https://docs.dagster.io/_apidocs/internals#dagster.PipelineRun"><code>PipelineRun</code></a> during execution and emit parent job (pipeline/job-level) RunEvents • execute steps using its helper (actual) executor • intercept yielded DagsterEvents, translate, and emit child job (op/step-level) RunEvents • yield original DagsterEvents</p> <p>Similar to openlienage-airflow, Dagster-specific <code>OpenLineageAdapter</code> can be implemented as a helper.</p> <p><strong>Generated OpenLineage Jobs &amp; RunEvents:</strong></p> <p>With the above initial work, the following jobs and events can be generated.</p> <p><strong>Dagster job (pipeline) run -&gt; OpenLineage parent job run</strong></p> <p>• During execution, <code>PipelineRun</code> is available from <code>PlanOrchestrationContext</code>, and we can get job/pipeline-level metadata from this PipelineRun and translate it to an OpenLineage event. • As an example, given <code>pipeline_run</code>, a RunEvent may look like,</p> <pre><code>RunEvent( eventType=event_type, eventTime=datetime.now().isoformat(), run=Run( runId=pipeline_run.run_id, ), job=Job( namespace=namespace, name=pipeline_run.pipeline_name, ), producer=PRODUCER, ) </code></pre> <p><strong>Dagster op (step) run -&gt; OpenLineage child job run</strong></p> <p>• Assuming "op" is equivalent to "step" at Dagster's low-level, we can identify op-related events from the returned DagsterEvents based on its <a href="https://docs.dagster.io/_apidocs/execution#dagster.DagsterEventType"><code>DagsterEventType</code></a>. For the initial implementation, the following types can be handled. • DagsterEventType.STEP<em>START • DagsterEventType.STEP</em>SUCCESS • DagsterEventType.STEP_FAILURE • Other types may be usable to extract additional metadata. • As an example, given <code>dagster_event</code>, a RunEvent may look like,</p> <pre><code>RunEvent( eventType=event_type, eventTime=datetime.now().isoformat(), run=self._build_run( namespace=namespace, run_id=run_id, # not associated with Dagster parent_run_id=dagster_event.logging_tags.get("run_id"), # pipeline run id parent_job_name=dagster_event.logging_tags.get("pipeline_name"), ), job=self._build_job( namespace=namespace, job_name=f"{dagster_event.pipeline_name}.{dagster_event.step_key}", ), producer=PRODUCER, )) </code></pre> <p><strong>How to Use in Dagster:</strong></p> <p>OpenLineageExecutor's ExecutorDefinition can be defined at the job-level like below.</p> <pre><code>from dagster import job, op from openlineage.dagster.executor import openlineage_executor @job(executor_def=openlineage_executor) def a_job(): @op def an_op(): pass an_op() </code></pre> <p>And the executor is configurable like below as Dagster's "executor config".</p> <pre><code>execution: config: openlienage: url: "https://..." api_key: "" namespace: "default" executor: multiprocess_executor: max_concurrent: 2 </code></pre> <hr /> <p>This proposal is based on limited research and understanding of Dagster, and there may be better alternatives as well that we are happy to discuss over and learn more so any feedback from the community is appreciated as we lay this initial block. Thank you for reviewing and looking forward to further discussions!</p>

<https://github.com/OpenLineage/OpenLineage/issues/489#issuecomment-1020718071|Comment on #489 [INTEGRATION][Dagster] Openlineage-Dagster Integration>
<p>Hi <a href="https://github.com/dalinkim">@dalinkim</a> - I'm a Dagster engineer, here to answer any questions you have. It would be awesome to see a Dagster-OpenLineage integration.</p> <p>A challenge with accomplishing this via a custom executor is that there are already a number of executor implementations that dagster users rely on. E.g. some use the K8s executor, some use the in-process executor, and some use the multiprocess executor. If you wanted to maintain that flexibility and also integrate via the executor, you'd need to write an openlineage version of each of those executors.</p> <p>The way that we've recommended implementing similar integrations in the past is via a Dagster sensor that tails the Dagster event log. Something like:</p> <pre><code>from dagster import sensor from dagster.core.storage.event_log.base import EventRecordsFilter @sensor def openlineage_sensor(context): event_records = context.instance.get_event_records( EventRecordsFilter(after_cursor=context.cursor), ascending=True, limit=5, ) for event_record in event_records: event_log_entry = event_record.event_log_entry storage_id = event_record.storage_id # write relevant contents of the event log entry to OpenLineage context.update_cursor(storage_id) </code></pre>

Hi all 👋

I have opened up an issue/proposal on getting datasets incorporated with the dagster integration. I would love to get some feedback and conversations going with the community on the proposed approach. Thanks!

<https://github.com/OpenLineage/OpenLineage/issues/583|#583 [INTEGRATION][DAGSTER] Add capability to include input and output datasets with the run event>
<p>With the initial work that has been done to integrate OpenLineage with Dagster (thanks <a href="https://github.com/dalinkim">@dalinkim</a>!), it seems like the next step should be to include a way to emit datasets along with the run events.</p> <p><strong>Proposal:</strong></p> <p>Introduce a way to transform Dagster assets or Outputs into InputDatasets and/or OutputDatasets to improve metadata collection.</p> <p><strong>Proposed Implementation:</strong></p> <p>I am no Dagster expert, but after working with the assets for a bit, there unfortunately doesn't seem to be a good way to automatically transform a generic <a href="https://docs.dagster.io/_apidocs/ops#dagster.AssetMaterialization">AssetMaterialization</a> or <a href="https://docs.dagster.io/_apidocs/ops#dagster.Output">Output</a> into an OpenLineage Dataset. However, specific OpenLineage assets can be manually created and yielded in a Dagster job/pipeline. Through the use of a wrapper/asset-generating function, a sufficiently unique AssetMaterialization with a defined structure can be yielded and later parsed into a dataset and added to the run event.</p> <p>&gt; Through the use of a wrapper/asset-generating function, a sufficiently unique AssetMaterialization with a defined structure can be created/yielded</p> <p>This function would take in an input/output dataset and create an AssetMaterialization—with an <a href="https://docs.dagster.io/_apidocs/ops#dagster.AssetKey">AssetKey</a> or unique structure that can be used to identify the asset as an OpenLineage dataset. That asset could then be yielded in a Dagster job/pipeline.</p> <p>&gt; and later parsed into a dataset</p> <p>An extra <code>elif</code> check can be added to the <a href="https://github.com/OpenLineage/OpenLineage/blob/8c15b9fe9dc88799d44fc3999f18a47909acb844/integration/dagster/openlineage/dagster/sensor.py#L145-L157">OpenLineage-Dagster sensor</a> to detect <a href="https://docs.dagster.io/_apidocs/execution#dagster.DagsterEventType.ASSET_MATERIALIZATION">when assets are yielded</a>. The asset can then be determined if it is a dataset by its AssetKey/structure and if it is, it can be parsed back into an input/output dataset by reversing the logic used in the above function.</p> <p>&gt; and added to the run event</p> <p>After the datasets have been parsed and are available in the sensor, they can be passed along to the emit methods so they can be added to the run event.</p> <p><strong>Example:</strong></p> <p>To emit a dataset, something similar to this would be added to a job/pipeline:</p> <pre><code>@op def an_op(): output_dataset = OutputDataset( namespace="example_namespace", name="example_name" ) asset = asset_generating_function(output_dataset) yield asset # or yield it directly in the function # asset_yielding_function(output_dataset) </code></pre> <p><strong>Limitations of this approach:</strong></p> <p>• In order to fit with other assets yielded by Dagster, the AssetKey should be 100% human readable (i.e. no UUIDs and have a hierarchy that makes sense). • The created AssetKey should be idempotent. A dataset with the same namespace/name should produce the same key so dagster can relate the assets to one another. A dataset is identified by its namespace and name and the asset should be too. • The AssetMaterialization (most likely the AssetKey) should contain a sufficiently unique identifier so other assets are not mistakenly identified as an OpenLineage dataset. • A good option for the AssetKey limitations could be ["OpenLineage", "InputDataset <em>OR</em> OutputDataset <em>OR</em> Dataset", "&lt;dataset.namespace&gt;", "&lt;dataset.name&gt;"]<br /> This approach is human readable, idempotent, and is a sufficiently unique identifier (assuming other assets beginning with OpenLineage &gt; Dataset are not being created elsewhere).</p>
